Masters – core configuration records

Masters hold all static reference data such as customers, vendors, banks, godowns and product definitions that are reused by purchase, inventory, sales and collection transactions.[file:89]

Secondary Customers

Retailer / outlet master

The Secondary Customers screen stores all retailer and outlet records to whom the distributor sells products from day to day.[file:89]

Each customer carries commercial attributes such as channel hierarchy, geography, payment mode, credit controls and contact details that drive billing and collection logic across the portal.[file:89]

How to create a new secondary customer
  1. From the top menu, navigate to Masters → Secondary Customers.[file:89]
  2. Click the “Create New Customer…” link to open the customer entry form.[file:89]
  3. Fill mandatory fields such as Customer Name, Channel Hierarchy, Geography, City, State and contact details.[file:89]
  4. Set credit controls like Outstanding limit, Credit days and number of credit bills, if applicable.[file:89]
  5. Save the record, then maintain Billing Address, Shipping Address and Beat mapping as required.[file:89]
Fields to create a Secondary Customer
  1. Customer Code – system‑generated; optionally override with the distributor’s own alphanumeric code used on reports and exports.[file:2]
  2. Customer Name – retailer / outlet shop name as it should appear on invoices and statements.[file:2]
  3. Channel Hierarchy – choose the retailer channel classification (e.g. General Trade, Key Accounts) from the HO‑defined list.[file:2]
  4. PIN Code, City, State – enter the shop’s postal PIN, city and state so geography and taxation work correctly.[file:2]
  5. Supply Chain / Distributor – defaults to the logged‑in distributor; keep the default unless specified by HO.[file:2]
  6. General / Value / Potential Classification – select applicable classifications to segment retailers by profile and potential.[file:2]
  7. Contact Person, Phone, Mobile, Email – capture primary contact details for follow‑ups and communication.[file:2]
  8. Market / Area / Geography – map the outlet to the correct market and geography codes used in targets and reports.[file:2]
  9. Customer Group – optionally tag the retailer into a customer group (e.g. Wholesaler, Semi‑Wholesale).[file:2]
  10. Credit Days / No. of Credit Bills / Credit Amount – numeric values controlling how long and how much credit is allowed before billing is blocked or warned.[file:2]
  11. Payment Mode & Cash Discount % – default payment mode and any fixed cash discount to be applied automatically.[file:2]
  12. Outstanding Validation – tick if the system should stop billing when outstanding exceeds the allowed credit controls.[file:2]
  13. Registration Date, ID Proof, Address Proof, Shop/Firm Type – registration and KYC details used during retailer approval.[file:2]

Once saved and approved, the Secondary Customer becomes available in Sales Order, Sales Invoice, Sales Return, Collection, schemes and all customer‑wise reports.[file:2]

Each vendor record holds identifiers such as Vendor Name, locality, city, state and credit term to control purchase and payment behaviour.[file:89]

  2. Use the “Create” (+) option to add a new vendor with name, code and address details.[file:89]

  4. Save the vendor; the record becomes selectable in Purchase Orders and Purchase Invoices.[file:89]

Van Master

Vehicle definition

Van Master defines each sales or delivery vehicle that can carry stock, including van code, description and status.[file:89]

Vans configured here are used in van allocation, van loading / unloading and van sales to track stock and billing by vehicle.[file:89]

How to create a Van master
  1. Open Masters → Van Master to view the list of vans.[file:2]
  2. Click the “Create New Van” (+) button to open the van entry form.[file:2]
  3. Verify or change the Van Code suggested by the system to match your internal van numbering.[file:2]
  4. Enter the vehicle registration in Van No and a clear description in Van Name (for example “TATA 407 – Route 1”).[file:2]
  5. Tick Status to activate the van so it appears in van loading, unloading and allocation screens.[file:2]
  6. Fill operational costing fields: Rate per Km, Van Load Capacity in Tons, Rate per Ton and Rate per Day as per your transport contracts.[file:2]
  7. Save the van record. It is now available for van loading, van sales and route‑wise performance analysis.[file:2]
Fields to create a Van
  1. Van Code – auto‑generated code that can be changed to follow the distributor’s internal van coding convention.[file:2]
  2. Van No – enter the vehicle registration / number plate so it is uniquely identifiable.[file:2]
  3. Van Name – friendly name or description, for example “TATA 407 – Route 1”.[file:2]
  4. Status – tick to activate; only active vans can be used in van‑related transactions.[file:2]
  5. Rate per Km – numeric per‑kilometre rate for transport costing.[file:2]
  6. Van Load Capacity in Tons – maximum carrying capacity in tons for planning and compliance.[file:2]
  7. Rate per Ton – charge per ton where costing is tonnage‑based.[file:2]
  8. Rate per Day – daily hire or running cost where applicable.[file:2]

Vans created here are used in van allocation, van loading/unloading and van sales to track stock and cost per vehicle.[file:2]

Gamification – KPI rules & Leaders Board

Performance contests & ranking

The Gamification screen defines KPI‑based game rules which measure how distributors or salesmen perform on key metrics such as number of orders, number of sales invoices and invoice or order value.[page:91]

Each row in the Gamification list represents one KPI rule with its own code, type, scope and start date, and provides a Leaders Board link to view rankings for that rule.[page:91]

Column Description
S.No Serial number of the gamification KPI definition in the list.
KPI Name Name of the KPI or contest (for example “Test Rep NOSI”, “GAM REP1”); clicking the name opens the KPI’s Leaders Board view.[page:91]
KPI Code Unique code auto‑generated for the KPI (for example GAMIFY239, GAMIFY232) and used as a technical identifier.[page:91]
KPI Type Metric being measured, such as Number of Sales Invoices, Number of Orders, Invoice Value, Order Value, Successfully converted Order Value, or Number of Orders Successfully converted to Invoices.[page:91]
Applicable Scope Level at which performance is calculated, typically Distributor Salesman (per salesman at a distributor) or Distributor (overall distributor).[page:91]
Start Date Date from which the KPI starts accumulating transactions; only orders/invoices on or after this date contribute to the Leaders Board.[page:91]
Leaders Board Link to open the ranking view for that KPI, showing who is leading based on the selected metric and scope.[page:91]
How to configure a Gamification KPI
  1. Navigate to Masters → Gamification to view the KPI list.[page:91]
  2. Click the “New” or “Create” (+) option to add a new KPI rule.
  3. Enter a clear KPI Name describing what is being measured (for example “Number of Orders – Feb 2026”).[page:91]
  4. Review or accept the system‑generated KPI Code (GAMIFYxxx) used internally for tracking.[page:91]
  5. Select the appropriate KPI Type, such as “Number of Sales Invoices”, “Number of Orders”, “Invoice Value”, “Order Value”, “Successfully converted Order Value” or “Number of Orders Successfully converted to Invoices”.[page:91]
  6. Set the Applicable Scope to “Distributor Salesman” if you want rankings per salesman, or “Distributor” if the contest is at distributor level.[page:91]
  7. Enter the Start Date from which the KPI should begin counting relevant orders and invoices.[page:91]
  8. Save the KPI; from this date, actual transactions matching the KPI Type and Scope will contribute to the Leaders Board.[page:91]

Once active, the KPI runs in the background, reading sales and order data so that performance can be shown as rankings, points or dashboards without changing the core transaction flow.[file:2][page:91]

Leaders Board – how ranking works
  1. From the Gamification list, click the KPI Name or Leaders Board link for the KPI you want to review.[page:91]
  2. The Leaders Board screen lists all applicable entities (for example all salesmen at a distributor) and shows their KPI value such as number of orders, number of invoices or value achieved.[page:91]
  3. Rows are sorted so the highest performers appear at the top, providing an immediate view of who is leading and who is lagging.[page:91]

The Leaders Board is used for contests, incentive programmes and performance reviews, helping management and field teams see where they stand against peers for a specific KPI.[file:2][page:91]

Every inventory transaction such as stock opening, GRN, van loading or stock transfer references a source and/or destination location from this list.[file:89]

Customer segmentation sequence

Priority & segment order

Customer segmentation sequence controls the ranking of retailer segments (for example A/B/C, GT/MT) used in reports, incentives and route planning.[file:2]

By maintaining a sequence number per segment, dashboards and exports can display higher‑value segments first.[file:2]

Fields used in Customer segmentation sequence and how to create
  1. Open the Customer segmentation sequence screen in Masters.[file:2]
  2. Select the Segment Type to configure (for example value classification, potential class or custom segment).[file:2]
  3. For each segment value (A, B, C, etc.), assign a numeric Sequence No where “1” is highest priority.[file:2]
  4. Optionally mark certain segments as Active so obsolete codes do not appear in new masters.[file:2]
  5. Save the sequence to apply the new ordering across reports and planning screens.[file:2]

These sequences are used in retailer lists, journey plans and performance reports to group and sort customers by business importance.[file:2]

Retailer sequence

Customer visit order

Retailer sequence defines the order in which retailers are visited on a route or beat, so that salesmen and reports follow a consistent call sequence.[file:2]

The sequence is usually configured once and then maintained whenever new outlets are added or existing outlets are re‑segmented.[file:2]

Fields used in Retailer sequence and how to create
  1. Open the Retailer sequence screen from Masters (or corresponding sequence menu where enabled).[file:2]
  2. Select the Beat / Route for which you want to define the call sequence.[file:2]
  3. Pick the Salesman responsible for that beat if the screen supports salesman‑wise sequences.[file:2]
  4. For each retailer, set a numeric Sequence No that reflects the planned visit order (1, 2, 3, …).[file:2]
  5. Save the sequence. The system will display the retailers sorted by Sequence No for that beat.[file:2]

The Retailer sequence is used when generating journey plans, mobile call lists and beat‑wise execution dashboards so that visits follow the pre‑agreed order.[file:2]

Transaction Series

Document numbering

Transaction Series controls document number patterns for transactions such as invoices, credit notes, receipts and stock documents.[file:89]

Administrators configure prefixes, running numbers and reset behaviour so each document type follows a structured numbering scheme.[file:89]

How to create a Transaction Series
  1. Go to Admin → Transaction Series to view all existing series.[file:2]
  2. Click the “Create New Transaction Series” (+) button.[file:2]
  3. Enter Transaction Series Name matching the transaction type (for example “Sales Invoice FY26”).[file:2]
  4. Provide a unique Transaction Series Code for easy identification.[file:2]
  5. Set Minimum Value and Maximum Value to define the numeric range of document numbers.[file:2]
  6. Choose Cycle Frequency (Daily, Monthly, Yearly, None) if the series should auto‑reset on a time basis.[file:2]
  7. Select the correct Transaction Type (PO, GRN, SI, SO, Collection, Payment, Sales Return, PI, etc.).[file:2]
  8. Define the numbering pattern in Pattern 1 / Pattern 2 using static text and dynamic tokens (DD, MM, YY, YYYY, nnnnn for running numbers).[file:2]
  9. Tick Mark as Default if this series should be used by default for the selected transaction type.[file:2]
  10. Enable Status to activate the series, then save.[file:2]

The Transaction Series is used by all corresponding documents (sales invoices, POs, collections, etc.) to generate structured and unique document numbers printed on outputs and used in exports.[file:2]

Custom Import / Universal / Import Datas

Bulk data uploads by category

The Custom Import module features 37 unique templates allowing users to download standard formats, fill them with bulk data, and upload them directly into the system. They are categorized below based on their business function.[page:97]

1. Master Data & Entity Mapping Imports

These templates are used to create or update core entities like retailers, sub-retailers, and route mappings.

  • Customer & Customer Update: Uploads or edits secondary customer details. Fields: Customer Code, Name, Channel, Geography, Mobile No, Credit terms. Usage: Bulk creation of the retail network.
  • Sub Retailer & Subretailer Update: Fields: Sub Retailer Code, Parent Customer Code, Contact details. Usage: Maps lower-tier stores reporting to a main distributor/wholesaler.
  • Beat Customer Relation / Beat Retailer Relation Import: Fields: Customer Code, Beat Code. Usage: Attaches retailers to specific routes so they can be billed.
  • Retailer Sequence: Fields: Beat Code, Customer Code, Sequence Number. Usage: Defines the exact order a salesman must visit shops on their route.
  • Salesman Master: Fields: Salesman Code, Name, Mobile, Status. Usage: Bulk creation of field staff.
  • Distributor Product Mapping / Name Mapping: Fields: System SKU Code, Distributor SKU Name/Code. Usage: Translates company SKU names into the distributor's local nomenclature.
2. Pricing & Commercials Imports

These templates manage custom pricing, margins, and credit configurations.

  • Retailer wise price: Fields: Customer Code, Product Code, Custom Rate. Usage: Overrides standard pricing for specific VIP retailers.
  • Special Price Setting: Fields: SKU Code, Start/End Date, Special Price. Usage: Uploads bulk promotional pricing for a defined period.
  • CDPrice (Cash Discount Price): Fields: SKU, Cash Discount %. Usage: Sets item-level cash discount parameters.
  • PTR Margin and Limit Import: Fields: SKU, Margin %. Usage: Sets standard Price To Retailer margins.
  • Credit Norm Mapping: Fields: Customer Code, Max Credit Days, Max Credit Amount. Usage: Bulk updates financial safety limits for retailers.
3. Salesforce Target & Journey Planning

Used to manage targets and daily working plans for the sales team.

  • MJP & MJP Update: Fields: Salesman Code, Date, Beat Code. Usage: Uploads the Monthly Journey Plan dictating where the salesman goes each day.
  • Salesman wise Target: Fields: Salesman Code, Month, Target Value/Volume. Usage: Initializes monthly KPIs.
  • Sales Forecast Import / Update: Fields: SKU, Month, Projected Qty. Usage: Uploads expected sales data to assist in purchase planning.
4. Inventory Management Imports

Templates to initialize or correct physical stock.

  • Stock Openings: Fields: Product Code, Godown, Batch, Expiry, Qty, Rate. Usage: Primarily used during system go-live to set day-one inventory.
  • Stock Adjustment: Fields: SKU, Godown, Adjustment Qty (+/-), Reason. Usage: Bulk upload of stock corrections following a physical audit.
  • Distributor StockNorm Map Import: Fields: SKU, Min Qty, Max Qty, Reorder Level. Usage: Defines automatic stock holding thresholds.
5. Transaction Imports (Sales & Purchase)

Uploads daily transactional data, bypassing manual entry screens.

  • Sales Order Import: Fields: Order Ref, Customer Code, SKU, Qty. Usage: Uploads bulk orders from third-party B2B apps.
  • E-Sales Invoice / E-Service Invoice Import: Fields: Invoice No, Date, Customer Code, Godown, SKU, Qty, Rate, Tax. Usage: Imports finalized billing data, immediately reducing stock.
  • Purchase Order Import: Fields: PO Ref, Vendor Code, SKU, Qty. Usage: Bulk orders placed to the principal company.
  • RPI (Received Purchase Invoice): Fields: Vendor Invoice No, SKU, Qty, Rate. Usage: Manually imports incoming vendor bills if the API integration is offline.
  • Sales Return (With/Without Ref) & E-Sales Return: Fields: Return Ref, Customer Code, SKU, Return Qty, Reason. Usage: Bulk uploads goods coming back from the market.
  • Purchase Return (With/Without Ref): Fields: Return Ref, Vendor Code, SKU, Qty, Reason. Usage: Bulk uploads items being sent back to the principal.
6. Financial & Collection Imports

Manages bulk financial adjustments and payments.

  • Debit Note / E-Credit Note Import: Fields: Note No, Party Code, Amount, Reason. Usage: Carries forward opening outstandings or bulk ledger corrections.
  • Salesman Wise Collection With Reference: Fields: Salesman Code, Customer Code, Invoice No, Amount Recd. Usage: Settles specific outstanding bills based on bank reconciliations.
  • Collection Without Reference: Fields: Customer Code, Amount Recd. Usage: Uploads bulk on-account advances where no specific invoice is targeted.

General Rules for all Custom Imports

  1. Template formatting: Click the specific link in the Import menu to download the exact `.csv` / `.xls` template. Do not alter column headers.
  2. Dates: Ensure Dates are strictly formatted as `yyyy-mm-dd`.
  3. Row ID: Ensure the `Rowid` column is sequentially numbered (1, 2, 3...) to help trace errors.
  4. Validation: After upload, review the Import Log immediately. If a row fails, the log shows the specific Rowid and the reason (e.g., "Invalid SKU Code"). Fix the data and re-upload only the failed rows.

Orders can be created from web or mobile, converted to invoices, and any post‑sale returns are captured with proper reasons and stock adjustments.[file:2]

Sales Order

Order capture

Sales Order (SO) captures orders from retailers, either as web orders or RSOs coming from mobile/SFA (shown as Mobile Order in the list).[file:2][page:94]

Orders can be maintained in Draft/Created/Published status and later converted to Sales Invoices as per business rules.[file:2][page:94]

How to create a Sales Order
  1. Go to Sales Management → Sales Order and click “Create New Sales Order…”.[page:94][file:2]
  2. System assigns a Transaction Number from the configured transaction series when saved.[file:2]
  3. Enter a Reference Number/Description (for example mobile order id or manual ref).[page:94][file:2]
  4. Select the Customer (retailer), Beat and Salesman will get autopopulated as per the customer beat mapping and salesman will get listed as per salesman beat mapping.[page:94][file:2]
  5. Enter the Sales Order Date (defaults to system date) and Order Type (Direct Order / Mobile Order as applicable).[page:94][file:2]
  6. In the item grid, for each product, capture:
    • Product and UOM.
    • Order Quantity and free quantity (if applicable).[file:2]
    • Rate, discounts and taxes as per pricing and schemes.[file:2]
  7. Save as Draft to edit later or set to Created/Published to make the order available for conversion to Sales Invoice.[file:2][page:94]

Customer, Beat, Salesman, Order Date and at least one product line with quantity and rate are mandatory to finalize a Sales Order.[file:2]

Fields in Sales Order
  1. Transaction Number – system‑generated document number (based on Transaction Series) assigned when saved.[file:2]
  2. Reference Number / Description – manual entry field to capture any external order ref (for example from mobile app or physical slip).[file:2]
  3. Customer (Retailer) – select the retail outlet. This auto‑populates billing and shipping details, channel and credit terms from the customer master.[file:2]
  4. Beat & Salesman – details will get autopopulated as per valid route mapping.[file:2]
  5. Order Type – The type of order:[page:94]
    • Direct Order / Mobile Order / CSLM Order / Retailer App Order / Import Order : standard flow for regular web or field orders.[file:2]
  6. Sales Order Date – defaults to today, used for reporting and pricing validity.[file:2]
  7. Product Grid (Item Details):
    • Product & UOM – choose the SKU and unit of measure.[file:2]
    • Order Quantity & Free Quantity – enter requested stock. For SAUDA, this draws from the contract balance.[file:2]
    • Rate, Discount & Tax – auto‑fetched from price lists. Can be edited if allowed, except for SAUDA orders which lock the rate.[file:2]
  8. Status – save as Draft to edit later, or Created/Published to finalize and allow conversion to Invoice.[file:2]

Where it is used: The approved Sales Order acts as the base document for generating the Sales Invoice. It updates pending order reports and salesman target achievements.[file:2]

Buttons and features in Sales Order list
